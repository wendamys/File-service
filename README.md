# File Service

Сервис скачивания и анализа каталога текстовых файлов через rate-limited API.

Каталог раздаётся сторонним API порциями: имена выдаются случайными группами по 3–9,
за один запрос можно скачать не более 3 файлов, а общее количество файлов сервер
не раскрывает. Частота запросов ограничена: при превышении лимита приходит `429`,
а настойчивому клиенту — блокировка на 30 минут (`403`).

## Возможности

- **Страница скачивания** — запуск процесса одной кнопкой, живой прогресс:
  время старта по Новосибирску, «получено N названий файлов, скачано M из N»,
  статус блокировки с обратным отсчётом, хвост лога.
- **Страница файлов** — список скачанного с сортировкой по времени и пагинацией,
  выбор файлов (точечно / все на странице / вообще все).
- **Расчёты** — встречаемость каждой цифры 0–9: общая по выборке и по каждому файлу.

## Стек

Python 3.14 · FastAPI · Jinja2 + vanilla JS · SQLAlchemy 2.0 + SQLite · requests · pytest

## Установка

```bash
uv sync
cp .env.example .env
```

В `.env` задать свой `CANDIDATE_ID` — к нему сервер привязывает прогресс скачивания.

## Запуск

```bash
uv run uvicorn app.main:app
```

Открыть `http://127.0.0.1:8000`.

## Тесты

```bash
uv run pytest
```

## Архитектура

```
app/
  config.py         # настройки из .env (pydantic-settings)
  exceptions.py     # типизированные ошибки API
  rate_limiter.py   # проактивный троттлинг (AIMD)
  api_client.py     # HTTP-клиент API: ретраи, backoff, разбор Retry-After
  zip_extractor.py  # распаковка ZIP с защитой от zip-slip
  storage.py        # БД метаданных: пагинация, сортировка, сверка с диском
  downloader.py     # оркестрация скачивания, переживание блокировки
  jobs.py           # фоновый job + потокобезопасное состояние для UI
  stats.py          # подсчёт цифр
  web/routes.py     # HTTP-роуты: страницы и JSON API
  main.py           # фабрика приложения
```

Слои разделены по зонам ответственности: клиент знает только про HTTP,
`Downloader` — только про порядок шагов, `Storage` — только про БД,
`JobManager` — мост между фоновым потоком и веб-слоем.

## Как сервис не попадает под блокировку

Блокировка привязана к **IP**, а не к идентификатору кандидата — служебная ручка
разблокировки принимает `client_ip`, тогда как прогресс хранится по `candidate_id`.
Менять `X-Candidate-Id`, чтобы обойти бан, бессмысленно. Поэтому ставка сделана
на то, чтобы `429` вообще не возникал.

**Проактивный троттлинг.** Все запросы проходят через `RateLimiter`, выдерживающий
минимальный интервал между ними (`REQUEST_INTERVAL`, по умолчанию 1.5 с). Схема
AIMD: на `429` интервал умножается на `BACKOFF_FACTOR` (клиент сам замедляется),
после серии успешных ответов — плавно возвращается к базовому.

**Ограниченные ретраи с джиттером.** Повторы на `429`, `5xx` и сетевые сбои
ограничены `MAX_RETRIES` — никаких бесконечных циклов. Пауза всегда берётся из
`Retry-After` (поддерживаются и секунды, и HTTP-дата) плюс случайный джиттер,
чтобы не долбить сервер ровно по границе разрешённого интервала.

**Экономия запросов.** Скачивание ограничено тройками (`maxItems: 3` в схеме API),
а вот отметка скачанных файлов верхнего предела не имеет. Поэтому порция из 9 имён
скачивается тремя запросами, но отмечается **одним**: 5 запросов вместо 7.

**Переживание блокировки.** `403` не роняет процесс: сервис показывает «заблокирован
до ЧЧ:ММ», прерываемо ждёт до разблокировки и продолжает с места остановки.
Уже скачанное не перекачивается — состояние восстанавливается из БД.

**Устойчивость к сбоям.** Файл отмечается на сервере только после того, как реально
записан на диск. Если отметка не прошла — на следующем запуске файл будет до-отмечен,
а не скачан заново. На старте БД сверяется с диском в обе стороны: пропавшие файлы
удаляются из индекса, найденные на диске — добавляются.

## Конфигурация

| Переменная | По умолчанию | Назначение |
|---|---|---|
| `BASE_URL` | `http://91.199.149.128:18001` | адрес API |
| `CANDIDATE_ID` | — | идентификатор кандидата |
| `DOWNLOADS_DIR` | `downloads` | куда складывать файлы |
| `DB_URL` | `sqlite:///file_service.db` | БД метаданных |
| `REQUEST_INTERVAL` | `1.5` | базовый интервал между запросами, с |
| `MAX_INTERVAL` | `15.0` | потолок интервала при backoff, с |
| `BACKOFF_FACTOR` | `1.5` | во сколько раз замедляться на `429` |
| `MAX_RETRIES` | `5` | лимит повторов одного запроса |
| `TIMEOUT` | `30.0` | таймаут HTTP-запроса, с |

Если в логах всё же появляются `429` — поднимите `REQUEST_INTERVAL`.
